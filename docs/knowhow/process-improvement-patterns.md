# プロセス改善パターン

## 知見

### /loop + /addf-dev によるバックログ自動消化

```bash
/loop 20m /addf-dev
```

20分間隔で `/addf-dev` を自動実行し、TODO バックログを連続的に消化する。

**動作**:
1. 各サイクルで TODO.md から未着手タスクを1つ選択
2. Plan 読み込み → 実装 → 品質ゲート → コミット → push
3. タスクがなくなると「全プラン完了、`/loop` を停止してください」と通知

**実績**: Plan 0018（ドキュメント同期 + セキュリティ修正）と Plan 0017（Bash 引数スコープ）を連続で完了。

**注意点**:
- 間隔は実装規模に合わせて調整（20m は小〜中規模タスク向き）
- cron の最小粒度は1分なので、秒単位の指定は切り上げ
- 7日後に自動期限切れ。`CronDelete` で早期停止可能

### ノウハウ蓄積ステップの欠落発見

Progress テンプレートの「作業中」ステップ4に「`/addf-knowhow` で記録する」と書いてあったが、品質ゲート後の完了処理フローには含まれていなかった。結果、18 Plan 分の実装で knowhow が1件も記録されなかった。

**原因**: 「作業中の知見記録」と「完了時の知見記録」が分離していた。品質ゲートの順序に knowhow ステップがないため、実行されなかった。

**修正**: Progress テンプレートのステップ13（完了処理内）に `/addf-knowhow` 実行を明示追加。以下の3観点で振り返る:
- コーディング: Go の実装パターン、mvdan.cc/sh の使い方、DSL パーサーのハマりポイント
- 品質ゲート: レビューエージェントから得た指摘パターン
- タスク総括: 計画と実装の乖離

**教訓**: チェックリストに入っていない作業は実行されない。「やるべき」と書くだけでなく、実行順序のフローに明示的に組み込む必要がある。

### Plan の即時作成パターン

会話の流れでユーザーがアイデアや要望を出したとき、その場で `docs/plans/` に Plan ファイルを作成し `TODO.md` に登録する運用:

- 「Bash コマンドの引数パスに対応する計画を次のプランに入れる」→ Plan 0017 即時作成
- セキュリティ + ドキュメントレビュー結果 → Plan 0018 即時作成
- 改善アイデア5個中3個採用 → Plan 0019/0020 即時作成

「後でやる」を Plan ファイルとして具体化することで、アイデアの揮発を防ぎつつ現在のタスクに集中できる。
