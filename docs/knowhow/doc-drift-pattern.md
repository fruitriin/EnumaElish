# ドキュメントドリフトのパターンと対策

## 知見

### 典型的なドリフトパターン

Plan 0018 のドキュメントレビューで Critical 4件が発見された。全て「実装済み機能がドキュメントで未実装扱い」のパターン。

| パターン | 例 |
|---|---|
| **ロードマップが更新されない** | Phase 11-15 が「将来の機能」のまま放置 |
| **printUsage に追加忘れ** | `detect`, `test`, `generate-rules` が `--help` に未記載 |
| **README のサブコマンド一覧が古い** | 新コマンドが日英両方で欠落 |
| **DSL リファレンスに新設定が未記載** | `workspace:` が settings ブロックの例に不在 |

### なぜ起きるか

- 実装が複数フェーズにまたがると、各フェーズでドキュメントの「一部」しか更新しない
- ロードマップの「将来」記述は実装完了後に誰かが明示的に「実装済み」に変更しない限り残る
- `printUsage()` やREADME のテーブルはコードと別ファイルなので自動では同期されない

### 対策

1. **品質ゲートに `addf-doc-review-agent` を組み込む** — 全タスク完了時にドキュメントとの乖離を自動検出。Progress テンプレートのステップ 7 に明記済み
2. **ロードマップは実装完了時に更新する** — Phase 完了時に roadmap.md のステータスを変更する
3. **`printUsage()` はコマンド追加時にチェックリストに入れる** — main.go の case 追加と printUsage 追加をセットで行う
